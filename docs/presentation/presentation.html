ADR-01

Monolito Modular vs Microserviços

Contexto Sistema novo · Dev solo · Domínio em maturação
Decisão Monolito modular com 4 módulos independentes
Ganho Deploy único · Sem latência entre módulos · Debug direto
Tradeoff Sem escala independente por módulo · Deploy isolado impossível
Conceitos: Monolito Modular · Screaming Architecture · SRP Arquitetural
ADR-02

Clean Architecture + EDA com RabbitMQ

Contexto Domínio de saúde exige regras isoladas · Eventos assíncronos necessários
Decisão Clean Architecture 4 camadas + EDA via RabbitMQ/AMQP
Ganho Domínio testável · Módulos desacoplados · Infra substituível
Tradeoff Complexidade adicional · Debug assíncrono mais difícil
Conceitos: Clean Architecture · EDA · Ports & Adapters · Inversão de Dependências
ADR-03

CQRS — Avaliado e Não Implementado

Contexto Exigência de avaliação do padrão — separar leitura e escrita
Decisão Não implementar — banco único, volume não justifica
Ganho Simplicidade · Um repositório por entidade · Menos código
Tradeoff Escala de leitura futura limitada
Resposta-chave: Casos de uso já separam intenção (create vs get) — caminho aberto para CQRS incremental

Arquiteto de Software

Como o código está organizado por dentro?

  • Clean Architecture com 4 camadas — dependências apontam para dentro
  • Entidades em Python puro — sem SQLAlchemy, sem FastAPI, sem RabbitMQ
  • Casos de uso dependem de contratos (ports/ABCs), nunca de implementações concretas
  • id do paciente ≠ user_id — identidade clínica desacoplada da identidade de acesso
  • Regra no domínio: paciente só vê os próprios exames → 403 Forbidden para outros
GetPatientExamsUseCase — se requester for patient e user_id não bater → UnauthorizedAccess
🔗
Arquiteto de Solução

Como os componentes conversam entre si?

  • API REST recebe requisições síncronas dos clientes
  • Ao criar exame ou laudo, publica evento no RabbitMQ via AMQP de forma assíncrona
  • 3 containers com Docker Compose + health checks — API só sobe após banco e broker prontos
  • Eventos carregam IDs suficientes para qualquer consumidor processar de forma independente
  • FakeBroker absorve falha do RabbitMQ com log — API continua funcionando

Arquiteto Enterprise

Por que essa arquitetura faz sentido para a empresa?

  • Custo operacional baixo — deploy único, sem orquestração complexa
  • Viável para postos de saúde públicos com equipe de TI reduzida
  • Controle de acesso por roles alinhado com LGPD e regulamentações de saúde
  • Módulos com fronteiras definidas permitem migração incremental para microserviços no futuro
